# スケジューリング設計（Phase 5.4 設計ドキュメント）

対象範囲（2026-07-06 ユーザー合意）:

1. 周期・位相（オフセット）設計とレート整合チェック
2. **実行順序の設計と「次の周期までに処理が終わるか」のデッドライン検証**
3. Executor / コールバックグループ設計
4. プロセス配置（composition）
5. RT 優先度・CPU 割当

## 1. モデル拡張（.arcsyn スキーマ）

### ノード（NodeDef）

```yaml
nodes:
  - id: n1
    label: SensorFusion
    period_ms: 50
    offset_ms: 10 # 位相オフセット（周期起点からのずれ。省略時 0）
    wcet_ms: 5.0 # 最悪実行時間の見積り [ms]（省略可。解析に使用）
```

- `wcet_ms` はユーザーが与える設計値（計測から更新していく想定）。
  未設定のノードはデッドライン解析の対象外（警告で明示）。

### プロジェクト（scheduling セクション）

```yaml
scheduling:
  processes: # プロセス配置。未所属ノードは各自単独プロセス
    - name: control_proc
      executor: single # single | multi
      threads: 4 # multi の場合のみ
      priority: 80 # RT 優先度（SCHED_FIFO）。省略可
      cpu_affinity: [2, 3] # CPU 割当。省略可
      nodes: [n1, n2] # 含めるノード id
```

- **実行順序の設計**は「同一プロセス（single executor）への配置 + オフセット」で表現する。
  single-threaded executor 内ではコールバックが直列化されるため、
  オフセットで起動位相をずらすことが実行順序の指定に相当する。

## 2. 静的解析（GUI でリアルタイム表示）

すべてフロントエンドの純関数（`src/utils/scheduling.ts`）で実装し、vitest で担保する。

### 2.1 解析モデルの仮定（明示）

- デッドライン = 周期（period_ms 以内に処理が完了すべき）
- single-threaded executor 内のコールバックは**非プリエンプティブに直列実行**される
- 解析は**安全側の上界**を用いる（厳密解析より悲観的だが、説明可能で健全）

### 2.2 チェック項目

| #   | 項目                             | 計算                                          | 判定                                 |
| --- | -------------------------------- | --------------------------------------------- | ------------------------------------ |
| A   | プロセス利用率                   | U = Σ (wcet_i / period_i)                     | single: U ≤ 1 / multi: U ≤ threads   |
| B   | 応答時間上界（デッドライン検証） | R_i = C_i + Σ\_{同一プロセスの他ノード j} C_j | R_i ≤ T_i                            |
| C   | レート整合                       | 接続ノード間の周期比                          | 整数比でなければ警告（位相ドリフト） |
| D   | データ鮮度                       | エッジの最悪データ齢 ≈ T_p + R_p              | 情報表示                             |
| E   | チェーン最悪レイテンシ           | 経路上の Σ (T_k + R_k) + R_末端               | 情報表示（source→sink）              |
| F   | オフセット衝突                   | 同一プロセス・同周期・同オフセット            | ずらす提案（ヒント）                 |
| G   | 配置妥当性                       | 多重所属 / 不明 id / 異言語混在プロセス       | エラー                               |

- B が「次の周期までに終わるか？」の検証。全ノードの WCET が揃っている
  プロセスのみ判定し、未設定は「WCET 未設定」の警告を出す。
- E が「どの順番でどれだけ遅れて届くか」の見える化。各エッジの
  サンプリング遅れ（最悪 T_p）+ 実行遅れ（R_p）の総和で近似する。

### 2.3 表示（スケジュール タブ）

- **プロセス定義**: 追加/削除、executor 種別・threads・priority・affinity、ノードのチェック選択
- **解析結果**: プロセスごとの利用率（%）と判定、ノードごとの R_i / T_i と判定、
  警告・ヒントの一覧。モデル変更のたびに自動再計算

## 3. コード生成への反映（5.4c）

| 設計要素             | Python                                               | C++                          | Rust                             | launch                                      |
| -------------------- | ---------------------------------------------------- | ---------------------------- | -------------------------------- | ------------------------------------------- |
| オフセット           | ワンショットタイマ → 周期タイマ起動                  | 同左                         | create_timer_oneshot → repeating | —                                           |
| プロセス配置         | `process_<name>.py`（複数ノードを1 executor に add） | `process_<name>.cpp`（同左） | 当面未対応（警告）               | Node → 生成 runner の ExecuteProcess        |
| executor 種別        | SingleThreadedExecutor / MultiThreadedExecutor       | rclcpp::executors::…         | —                                | —                                           |
| RT 優先度 / affinity | —                                                    | —                            | —                                | `prefix="chrt -f <prio> taskset -c <cpus>"` |

- 異言語ノードの同一プロセス配置は不可（解析 G でエラー）
- rclcpp_components による本格 composition は将来課題とし、
  まず「生成した main で複数ノードを同一 executor に載せる」方式を取る
